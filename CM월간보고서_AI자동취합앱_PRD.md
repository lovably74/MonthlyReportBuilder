# PRD: CM단 월간보고서 AI 자동취합·자동생성 애플리케이션 (서버-클라이언트 구조)

문서 버전: v2.0  
작성일: 2026-06-15  
대상 프로젝트: CM단 월간보고서 자동생성 윈도우 설치형 애플리케이션  
주요 사용 도구: Kiro, 로컬 LLM(서버), HWPX 템플릿 엔진  
기본 전제: 동일 네트워크망 내 서버-클라이언트 구조로 운영. 인터넷 없이 동작해야 함  
배포 방식: 윈도우 전용 설치형 (서버/클라이언트 선택 설치)

---

## 1. 문서 목적

본 문서는 로컬 PC에서 특정 폴더를 지정하면 해당 폴더 내 공문, 회의록, 공정자료, 안전자료, 품질자료, 검측자료, 자재승인자료, 기술검토자료, 기성자료, 공사사진 등을 자동으로 취합·분류·분석하여 CM단 월간보고서 HWPX 양식에 자동 반영하는 애플리케이션의 제품 요구사항 정의서(PRD)이다.

본 PRD는 Kiro를 이용한 개발을 전제로 하며, Kiro가 기능 단위로 작업을 분해하고 구현할 수 있도록 기능 요구사항, 비기능 요구사항, 데이터 구조, 화면 구조, 예외처리, 테스트 기준, 산출물 기준을 상세히 정의한다.

본 애플리케이션은 단순한 AI 문서작성기가 아니라, 월간보고서 작성을 위한 로컬 기반 문서 취합·분류·대장 생성·사진대지 생성·HWPX 문서 조립 시스템이다.

---

## 2. 배경 및 문제 정의

CM단 월간보고서는 매월 반복적으로 작성되지만, 보고서 작성 과정에서 다음과 같은 문제가 발생한다.

1. 여러 폴더에 흩어진 문서를 사람이 직접 찾아야 한다.
2. 공문, 회의록, 공정자료, 안전자료, 검측자료, 자재승인자료 등을 수작업으로 분류해야 한다.
3. 문서번호, 일자, 제목, 수신처, 처리결과 등을 다시 엑셀 또는 한글 표로 옮겨야 한다.
4. 사진대지를 수작업으로 구성해야 한다.
5. 월간보고서 양식의 대장 위치, 갑지 위치, Appendix 위치에 자료를 수동으로 삽입해야 한다.
6. 숫자, 공정률, 기성금액 등은 정확성이 중요한데, 사람이 복사·붙여넣기 과정에서 오류를 만들 수 있다.
7. 인터넷이 차단된 현장 또는 보안 환경에서는 클라우드 AI 사용이 어렵다.
8. 여러 담당자가 동일 LLM 자원을 사용해야 하지만, 개별 PC마다 LLM을 설치·관리하는 것은 비효율적이다.

따라서 본 애플리케이션은 동일 네트워크망 내에서 서버-클라이언트 구조로 운영되며, 서버가 LLM 자원과 중앙 데이터를 관리하고, 클라이언트는 문서 취합·분류·보고서 생성 UI를 제공하는 구조로 설계한다.

## 2-1. 서버-클라이언트 아키텍처 요구사항

### 2-1.1 배포 및 설치 구조

본 애플리케이션은 윈도우 전용 설치형으로 배포한다. 설치 시 사용자는 다음 중 하나를 선택한다.

1. **서버 설치** — LLM 엔진, 중앙 DB, 서버 발견 서비스, API 서버를 포함
2. **클라이언트 설치** — UI 애플리케이션, 로컬 작업폴더 관리, 서버 연결 모듈을 포함

### 2-1.2 서버 역할 및 구성

서버는 동일 네트워크망 내에서 다음 역할을 수행한다.

| 역할 | 설명 |
|---|---|
| LLM 호스팅 | Ollama/LM Studio 등 로컬 LLM 실행 및 API 제공 |
| 중앙 DB | SQLite 또는 PostgreSQL 기반 설정·작업 이력 관리 |
| 서버 발견 서비스 | mDNS/DNS-SD 기반 자동 서버 탐색 응답 |
| 서버 인증 | 고유 Server-ID 기반 인증 (UUID, IP 변경에 무관) |
| API 서버 | FastAPI 기반 REST API 제공 |
| 파일 공유 | 템플릿, 샘플 문서 등 공유 자원 관리 |

### 2-1.3 클라이언트 역할 및 구성

클라이언트는 다음 역할을 수행한다.

| 역할 | 설명 |
|---|---|
| UI | Tauri + React 기반 데스크톱 UI |
| 로컬 작업 | 파일 스캔, 복사, 작업폴더 관리 |
| 서버 통신 | REST API를 통한 서버 기능 호출 |
| LLM 요청 | 서버의 LLM 자원을 원격으로 활용 |
| 로컬 캐시 | 서버 응답 캐시, 오프라인 대비 |

### 2-1.4 서버 발견 및 인증 메커니즘

#### 서버 고유 식별

- 서버는 최초 설치 시 **Server-ID (UUID v4)**를 생성하여 저장한다.
- Server-ID는 서버를 재설치하지 않는 한 변경되지 않는다.
- IP 주소가 변경되더라도 Server-ID로 동일 서버임을 확인할 수 있다.

#### 자동 서버 발견 (mDNS/DNS-SD)

- 서버는 `_cm-report-server._tcp.local.` 서비스를 mDNS로 광고한다.
- 광고 정보에는 Server-ID, 서버명, API 포트, 서버 버전을 포함한다.
- 클라이언트는 동일 네트워크 대역에서 mDNS 쿼리로 서버를 자동 탐색한다.

#### 서버 등록 절차

1. 클라이언트 설치 완료 후 첫 실행 시 자동으로 서버 탐색을 시작한다.
2. 동일 네트워크에서 서버가 발견되면 서버 정보(서버명, Server-ID, IP, 포트)를 표시한다.
3. 사용자가 확인하면 Server-ID를 로컬에 등록한다.
4. 이후 IP가 변경되어도 Server-ID로 서버를 재탐색한다.
5. 서버를 찾을 수 없으면 수동 IP 입력 옵션을 제공하되, 연결 시 Server-ID를 검증한다.

#### 서버 미발견 시 처리

- 클라이언트 설치 후 서버가 탐색되지 않으면 다음 메시지를 표시한다:
  - "동일 네트워크에서 CM 월간보고서 서버를 찾을 수 없습니다. 관리자에게 서버 설치를 요청해 주세요."
- 서버 미연결 상태에서는 LLM 요약 기능은 비활성화하되, 로컬 작업(파일 스캔, 복사, 수동 분류)은 가능하도록 한다.

### 2-1.5 서버 중복 설치 방지

- 서버 설치 프로그램은 실행 시 동일 네트워크 내 기존 서버를 mDNS로 탐색한다.
- 기존 서버가 발견되면 다음 메시지를 표시하고 설치를 중단한다:
  - "동일 네트워크에 이미 CM 월간보고서 서버([서버명], [IP])가 설치되어 있습니다. 하나의 네트워크에는 하나의 서버만 운영할 수 있습니다."
- 기존 서버가 응답하지 않는 경우(종료/장애 상태)에만 새 서버 설치를 허용한다.

### 2-1.6 LLM 서버 자원 활용

- 클라이언트는 자체적으로 LLM을 실행하지 않는다.
- AI 요약, 문서 분류 보조, 문체 변환 등 LLM이 필요한 모든 작업은 서버의 LLM API를 호출한다.
- 서버는 LLM 요청 큐를 관리하여 다수 클라이언트의 동시 요청을 순차 처리한다.
- 클라이언트는 LLM 작업의 진행 상태를 폴링 또는 WebSocket으로 확인할 수 있다.
- 서버 연결이 끊어진 경우 LLM 기능은 비활성화되며, 사용자에게 서버 연결 복구 안내를 표시한다.

### 2-1.7 네트워크 통신 보안

- 서버와 클라이언트 간 통신은 동일 네트워크망(로컬 LAN) 내에서만 허용한다.
- 인터넷으로의 아웃바운드 트래픽은 여전히 발생하지 않는다.
- 서버 API는 Server-ID 기반 토큰 인증을 적용한다.
- 추후 TLS 적용을 위한 인터페이스를 준비하되, MVP에서는 HTTP를 사용한다.

따라서 본 애플리케이션은 로컬 PC에서 단독으로 동작하면서, 정해진 월간보고서 양식과 문서 샘플 기준에 따라 필요한 자료만 취합하고, 대장과 사진대지를 자동 생성하여 월간보고서 초안을 생성하는 것을 목표로 한다.

## 3. 핵심 원칙

### 3.1 AI 역할 제한 원칙

AI는 월간보고서의 모든 내용을 임의로 생성하지 않는다.

AI의 역할은 다음으로 제한한다.

- 문서 분류 보조
- 문서 요약
- 회의록 주요 안건 정리
- 공정 지연 사유 문장화
- 안전·품질 이슈 요약
- Summary 초안 생성
- 발주청 보고용 문체 변환

다음 항목은 AI가 추측하거나 임의 생성하면 안 된다.

- 문서번호
- 공문 발신일자 및 수신일자
- 공정률
- 계약금액
- 기성금액
- 검측일자
- 승인일자
- 업체명
- 수신처
- 처리결과
- 사진 촬영일자

위 항목은 원본 문서, 파일 메타데이터, 엑셀 자료, 사용자가 입력한 값 또는 추출 규칙 기반으로만 생성한다.

### 3.2 네트워크 내 독립 운영 원칙

외부 인터넷 연결이 없어도 동일 네트워크망 내에서 서버-클라이언트 구조로 다음 기능은 정상 동작해야 한다.

서버 측:
- LLM 모델 실행 및 API 제공
- 중앙 DB 관리
- 서버 발견 서비스 (mDNS)
- 프로필, 템플릿, 샘플 문서 중앙 관리
- LLM 요청 큐 관리

클라이언트 측:
- 환경설정 조회·수정 (서버 연동)
- 월간보고서 템플릿 등록 (서버에 업로드)
- 샘플 문서 등록 (서버에 업로드)
- 폴더 스캔 (로컬)
- 파일 복사 (로컬 작업폴더)
- 문서 분류 (키워드 기반은 로컬, LLM 보조는 서버)
- 텍스트·표 추출 (로컬)
- LLM 요약 (서버 호출)
- 대장 생성 (로컬)
- 사진대지 생성 (로컬)
- HWPX 결과 생성 (로컬)
- 로그 및 이력 저장 (로컬 + 서버 동기화)

클라우드 LLM API, 외부 OCR API, 외부 문서 변환 API는 기본 기능에 포함하지 않는다. 향후 옵션으로만 확장 가능하도록 인터페이스를 분리한다.

### 3.3 원본 보존 원칙

자동 취합 시 원본 파일은 절대 수정하지 않는다.

모든 분석과 가공은 작업폴더에 복사된 파일을 기준으로 수행한다.

### 3.4 검토 후 생성 원칙

자동 취합 결과는 곧바로 최종본으로 확정하지 않는다.

사용자는 최종 HWPX 생성 전 다음 항목을 검토할 수 있어야 한다.

- 취합 대상 파일 목록
- 제외 파일 목록
- 분류 결과
- 대장 데이터
- AI 요약문
- 사진대지 구성
- 누락 항목
- 오류 및 경고

## 5. 주요 사용 시나리오

### 5.1 최초 환경설정 시나리오

1. 사용자가 애플리케이션을 실행한다.
2. 환경설정 메뉴로 이동한다.
3. 월간보고서 샘플 HWPX 양식을 등록한다.
4. 공문, 회의록, 공정자료, 안전자료 등 문서 유형별 샘플 파일을 등록한다.
5. 작업폴더 루트 경로를 지정한다.
6. 최종 월간보고서 저장 위치와 파일명 규칙을 지정한다.
7. 대장 제목과 사진대지 표기 방식을 지정한다.
8. HWPX 양식의 데이터 삽입 위치를 지정한다.
9. 저장한다.

### 5.2 월간보고서 자동취합 시나리오

1. 사용자가 자동취합 메뉴로 이동한다.
2. 보고년월을 입력한다.
3. 취합 기준일을 시작일~종료일로 지정한다.
4. 저장된 파일 루트경로를 선택한다.
5. 실행 버튼을 누른다.
6. 애플리케이션이 해당 기간의 파일을 검색한다.
7. 환경설정의 샘플 문서와 유사한 파일만 분류·취합한다.
8. 작업폴더 하위에 보고월 기준 신규 폴더를 생성한다.
9. 취합 대상 파일을 작업폴더로 복사한다.
10. 동일 파일이 이미 있으면 덮어쓰기, 건너뛰기, 이름 변경 중 선택하게 한다.
11. 사용자는 선택을 단일 파일에만 적용하거나 동일 작업 전체에 적용할 수 있다.
12. 애플리케이션이 문서별 대장 데이터를 생성한다.
13. AI가 요약문을 생성한다.
14. 사진대지를 구성한다.
15. 사용자가 취합 결과를 검토한다.
16. 최종 월간보고서 HWPX를 생성한다.

# Part A. 환경설정 기능 요구사항

## 7. 환경설정 개요

환경설정은 자동취합과 보고서 생성을 위한 기준정보를 등록하는 영역이다.

환경설정은 한 번만 설정하는 것이 아니라, 현장별·양식별·발주처별로 여러 프로필을 저장할 수 있어야 한다.

---

## 8. 환경설정 - 프로필 관리

### 8.1 기능 설명

사용자는 여러 개의 설정 프로필을 만들 수 있어야 한다.

예시:

- 기본 CM 월간보고서 프로필
- A현장 발주처 제출용 프로필
- B현장 내부보고용 프로필
- 안전관리 강화형 프로필

### 8.2 필수 입력값

| 항목 | 설명 | 필수 여부 |
|---|---|---|
| 프로필명 | 설정 프로필 이름 | 필수 |
| 설명 | 프로필 설명 | 선택 |
| 기본 사용 여부 | 기본 프로필로 사용할지 여부 | 선택 |
| 생성일 | 자동 저장 | 자동 |
| 수정일 | 자동 저장 | 자동 |

### 8.3 기능 요구사항

- 프로필 생성
- 프로필 수정
- 프로필 복사
- 프로필 삭제
- 기본 프로필 지정
- 프로필 내보내기
- 프로필 가져오기

## 9. 월간보고서 샘플양식 등록

### 9.1 기능 설명

사용자는 월간보고서 샘플양식을 등록할 수 있어야 한다.

등록 대상 파일 형식은 다음과 같다.

| 형식 | 지원 수준 | 설명 |
|---|---|---|
| HWPX | 필수 지원 | 기본 권장 형식 |
| HWP | 선택 지원 | 한글 설치 또는 SDK 필요 |
| DOCX | 선택 지원 | 향후 확장 |

MVP에서는 HWPX를 기본 지원한다.

### 9.2 등록 절차

1. 사용자가 한글파일 불러오기 버튼을 클릭한다.
2. HWPX 또는 HWP 파일을 선택한다.
3. 파일을 앱 내부 템플릿 저장소로 복사한다.
4. 템플릿 내부의 치환 태그 또는 삽입 앵커를 분석한다.
5. 분석 결과를 화면에 표시한다.
6. 사용자가 각 태그의 용도를 지정한다.
7. 템플릿 버전을 저장한다.

### 9.3 템플릿 태그 예시

월간보고서 양식에는 다음과 같은 태그 또는 앵커를 사용할 수 있다.

```text
PROJECT_NAME
WRITE_YYMM
SUMMARY_TEXT
TFA_LIST
IRR_LIST
TR_LIST
DN_LIST
NCR_LIST
FI_LIST
SCAR_LIST
TFA_IMG
IRR_IMG
DN_IMG
NCR_IMG
FI_IMG
SCAR_IMG
PHOTO_SITE_VIEW
PHOTO_PROGRESS
```

### 9.4 요구사항

- HWPX 파일 등록 가능
- 등록한 템플릿 원본 보존
- 템플릿 버전 관리
- 템플릿 내 텍스트 태그 자동 탐색
- 사용자가 직접 태그를 추가·수정 가능
- 템플릿 미리보기 제공
- 템플릿 분석 실패 시 상세 오류 표시

### 9.5 예외처리

| 상황 | 처리 방식 |
|---|---|
| HWP 파일 선택 | HWPX 변환 가능 여부 확인. 변환 불가 시 HWPX 저장 후 재등록 안내 |
| 태그 없음 | 수동 매핑 방식으로 진행 가능 |
| 손상된 HWPX | 등록 실패 및 오류 로그 저장 |
| 동일 템플릿 등록 | 새 버전으로 등록할지 확인 |

## 10. 취합 문서 샘플 등록

### 10.1 기능 설명

사용자는 자동취합 시 기준이 되는 문서 샘플을 등록할 수 있어야 한다.

문서 샘플은 실제 취합 시 해당 문서와 유사한 파일만 자동으로 취합하기 위한 기준으로 사용한다.

### 10.2 문서 유형

기본 문서 유형은 다음과 같다.

| 코드 | 문서 유형 | 월간보고서 반영 위치 |
|---|---|---|
| LETTER_OUT | 발신공문 | 4.3 문서 처리현황, 발신공문List |
| LETTER_IN | 수신공문 | 4.3 문서 처리현황 |
| MEETING | 회의록 | 4.2 회의 진행현황, Summary |
| PROGRESS | 공정관련문서 | 3.1 월간 공정진행현황, 3.4 공정보고 |
| SAFETY | 안전관련문서 | 3.2 안전관리 현황, 4.10 안전관리 업무 |
| QUALITY | 품질관련문서 | 품질 관련 Summary, Appendix |
| INSPECTION | 검측관련문서 | 4.5 검측요청 및 결과통보 |
| MATERIAL | 자재승인 | 4.4 자재 및 자재공급원 승인업무 |
| TECH_REVIEW | 기술검토 | 4.7 건설사업관리단 검토의견서 |
| PAYMENT | 기성 | 4.9 기성금 지급현황 |
| PHOTO | 공사사진 | 5.1, 5.2 사진대지 |
| DN | 결함통보서 | 4.8.1 결함통보서 |
| NCR | 부적합보고서 | 4.8.2 부적합 통보 |
| FI | 작업지시서 | 4.8.3 작업지시서 |
| SCAR | 안전시정지시서 | 4.10.3 안전시정지시서 |

### 10.3 샘플 등록 방식

문서 유형별로 다음 정보를 등록한다.

| 항목 | 설명 |
|---|---|
| 문서 유형 | 공문, 회의록, 공정자료 등 |
| 샘플 파일 | 유사도 판단 기준 파일 |
| 대표 키워드 | 파일명 또는 본문에서 자주 등장하는 키워드 |
| 제외 키워드 | 해당 문서로 분류하면 안 되는 키워드 |
| 지원 확장자 | hwpx, hwp, docx, xlsx, pdf, jpg 등 |
| 유사도 기준값 | 자동 분류 임계값 |
| 날짜 추출 방식 | 파일명, 문서 본문, 파일 수정일 중 우선순위 |
| 대장 생성 여부 | 해당 유형의 대장을 생성할지 여부 |
| Appendix 삽입 여부 | 원문 이미지 또는 PDF 이미지를 Appendix에 넣을지 여부 |

### 10.4 유사 문서 판단 로직

유사 문서 판단은 다음 4단계를 조합한다.

1. 폴더명 기반 판단
2. 파일명 키워드 기반 판단
3. 본문 키워드 기반 판단
4. 로컬 임베딩 유사도 기반 판단

MVP에서는 1~3번을 우선 구현하고, 이후 4번을 확장한다.

### 10.5 문서 유형별 설정 예시

```yaml
document_types:
  LETTER_OUT:
    name: "발신공문"
    folder_keywords: ["공문", "발신"]
    filename_keywords: ["공문", "발신", "제출", "통보"]
    body_keywords: ["수신", "참조", "제목", "문서번호"]
    exclude_keywords: ["회의록", "사진"]
    extensions: ["hwpx", "hwp", "docx", "pdf"]
    similarity_threshold: 0.78
    generate_ledger: true
    append_original_image: true
```

## 11. 작업폴더 생성 위치 지정

### 11.1 기능 설명

사용자는 자동취합 작업 시 생성될 작업폴더의 루트 위치를 지정할 수 있어야 한다.

### 11.2 폴더 생성 규칙

사용자가 지정한 작업폴더 루트 아래에 보고월 기준으로 하위 폴더를 자동 생성한다.

예시:

```text
D:\CM_Monthly_Report_Work
 └─ 2026-07_월간보고서
     ├─ 00_원본복사
     ├─ 01_분류결과
     ├─ 02_추출텍스트
     ├─ 03_대장데이터
     ├─ 04_AI요약
     ├─ 05_사진대지
     ├─ 06_생성문서
     └─ logs
```

### 11.3 요구사항

- 작업폴더 루트 경로 지정 가능
- 폴더가 없으면 자동 생성
- 권한이 없으면 오류 표시
- 보고월별 하위 폴더 자동 생성
- 동일 보고월 폴더가 있으면 이어서 작업 또는 새 작업 생성 선택 가능

## 12. 대장 타이틀 및 사진대지 표기방식 설정

### 12.1 기능 설명

취합되는 문서들의 대장을 생성하기 위한 제목과 컬럼명을 환경설정에서 지정할 수 있어야 한다.

### 12.2 대장 설정 항목

문서 유형별로 다음 항목을 설정한다.

| 항목 | 설명 |
|---|---|
| 대장명 | 예: 발신공문List, 승인요청서, 검측요청 및 결과통보 |
| 번호 컬럼명 | No, 순번 등 |
| 일자 컬럼명 | 일자, 승인일, 요청일 등 |
| 문서번호 컬럼명 | 문서번호 |
| 제목 컬럼명 | 제목, 내용, 주요 내용 등 |
| 수신처/발신처 컬럼명 | 수신처 또는 발신처 |
| 결과 컬럼명 | 완료, 진행중, 조치완료 등 |
| 정렬 기준 | 일자순, 문서번호순, 파일명순 |
| 중복 처리 기준 | 문서번호, 제목+일자, 파일 해시 |

### 12.3 사진대지 표기방식

사진대지는 다음 설정을 지원한다.

| 항목 | 설명 |
|---|---|
| 사진 배열 | 1페이지 2장, 4장, 6장 |
| 사진 제목 표시 | 표시/미표시 |
| 촬영일자 표시 | 표시/미표시 |
| 위치 표시 | 표시/미표시 |
| 설명 표시 | 표시/미표시 |
| 파일명 표시 | 표시/미표시 |
| 사진 비율 | 원본비율 유지, 영역 채우기, 영역 맞춤 |
| 전경사진 구분 | 파일명/폴더명/사용자 지정 |
| 공사진행사진 구분 | 파일명/폴더명/사용자 지정 |

### 12.4 사진 캡션 예시

```text
[2026.07.31] 현장 동측 전경
[2026.07.31] 지상 3층 철근 배근 현황
[2026.07.30] 기계실 배관 설치 현황
```

## 13. 샘플양식 내 데이터 위치 지정

### 13.1 기능 설명

사용자는 등록한 월간보고서 양식에서 데이터가 삽입될 위치를 지정할 수 있어야 한다.

위치 지정 방식은 다음 2가지를 지원한다.

1. 태그 기반 위치 지정
2. 수동 앵커 위치 지정

### 13.2 태그 기반 위치 지정

HWPX 양식에 미리 태그를 넣어 둔다.

예시:

```text
PROJECT_NAME
WRITE_YYMM
TFA_LIST
IRR_LIST
DN_LIST
NCR_LIST
FI_LIST
SCAR_LIST
PHOTO_PROGRESS
```

애플리케이션은 HWPX 내부 XML에서 해당 태그를 찾아 실제 데이터로 치환한다.

### 13.3 수동 앵커 위치 지정

태그가 없는 양식의 경우, 사용자가 화면에서 삽입 위치를 수동으로 지정한다.

MVP에서는 태그 기반 위치 지정을 우선 구현한다.

### 13.4 위치 매핑 설정 예시

```yaml
template_mapping:
  PROJECT_NAME:
    type: text
    source: basic.project_name
  WRITE_YYMM:
    type: text
    source: basic.report_yymm
  TFA_LIST:
    type: table
    source: ledgers.tfa
  IRR_LIST:
    type: table
    source: ledgers.irr
  TFA_IMG:
    type: appendix_images
    source: appendix.tfa_images
  PHOTO_PROGRESS:
    type: photo_board
    source: photos.progress
```

## 14. 최종 월간보고서 저장위치 및 파일명 규칙

### 14.1 기능 설명

사용자는 최종 월간보고서가 저장될 위치와 파일명 규칙을 지정할 수 있어야 한다.

### 14.2 저장 위치

- 기본 저장 위치 지정 가능
- 폴더가 없으면 자동 생성
- 현장별 저장 폴더 사용 가능
- 보고월별 하위 폴더 자동 생성 가능

### 14.3 파일명 규칙

파일명 규칙은 변수 기반으로 설정할 수 있어야 한다.

지원 변수 예시:

```text
{PROJECT_NAME}
{REPORT_YYYY}
{REPORT_MM}
{REPORT_YYMM}
{ROUND}
{CREATED_YYYYMMDD}
{CREATED_HHMM}
```

파일명 예시:

```text
{PROJECT_NAME}_건설사업관리_월간보고서_{REPORT_YYYY}년{REPORT_MM}월.hwpx
```

결과 예시:

```text
OO신축공사_건설사업관리_월간보고서_2026년07월.hwpx
```

### 14.4 중복 파일 처리

최종 저장 위치에 동일 파일명이 존재할 경우 다음 옵션을 제공한다.

1. 덮어쓰기
2. 건너뛰기
3. 파일명 뒤에 버전 자동 추가
4. 사용자에게 매번 확인

# Part B. 자동취합 기능 요구사항

## 15. 자동취합 개요

자동취합은 사용자가 지정한 기준정보에 따라 원본 자료 폴더에서 필요한 파일만 찾고, 작업폴더로 복사한 뒤, 환경설정 기준에 따라 문서를 자동 분류·분석·취합하는 기능이다.

---

## 16. 자동취합 기본정보 입력

### 16.1 입력 항목

| 항목 | 설명 | 필수 여부 |
|---|---|---|
| 보고년월 | 월간보고서 대상 월 | 필수 |
| 취합 기준 시작일 | 자료 취합 시작일 | 필수 |
| 취합 기준 종료일 | 자료 취합 종료일 | 필수 |
| 저장된 파일 루트경로 | 원본 자료가 있는 최상위 폴더 | 필수 |
| 프로젝트명 | 월간보고서 표지 및 본문에 사용 | 필수 |
| 차수 | 제1차, 제2차 등 | 선택 |
| 적용 환경설정 프로필 | 사용할 설정 프로필 | 필수 |

### 16.2 기본값

- 보고년월: 현재일 기준 전월을 기본값으로 설정 가능
- 취합 시작일: 보고월 1일
- 취합 종료일: 보고월 말일
- 작업폴더: 환경설정의 기본 작업폴더
- 저장 위치: 환경설정의 기본 저장 위치

---

## 17. 파일 검색 및 필터링

### 17.1 검색 대상

사용자가 지정한 루트경로 하위의 모든 파일을 재귀적으로 검색한다.

### 17.2 지원 확장자

MVP 필수 지원:

```text
.hwpx
.docx
.xlsx
.pdf
.jpg
.jpeg
.png
```

선택 지원:

```text
.hwp
.xls
.pptx
.txt
.csv
.tif
.tiff
```

### 17.3 날짜 필터링 기준

날짜 판단 우선순위는 다음과 같다.

1. 파일명에 포함된 날짜
2. 문서 본문에 포함된 작성일자
3. 사진 EXIF 촬영일자
4. 파일 수정일자
5. 파일 생성일자

사용자는 환경설정에서 우선순위를 변경할 수 있다.

### 17.4 파일명 날짜 패턴

다음 패턴을 인식해야 한다.

```text
2026-07-15
2026.07.15
20260715
26.07.15
07월15일
2026년 7월 15일
```

## 18. 작업폴더 생성 및 파일 복사

### 18.1 작업폴더 생성

자동취합 실행 시 다음 경로에 신규 작업폴더를 생성한다.

```text
{WORK_ROOT}\{REPORT_YYYY}-{REPORT_MM}_월간보고서
```

동일 폴더가 이미 존재할 경우 다음 중 선택한다.

1. 기존 폴더 사용
2. 새 버전 폴더 생성
3. 기존 폴더 삭제 후 새로 생성
4. 취소

새 버전 폴더 예시:

```text
2026-07_월간보고서_v2
2026-07_월간보고서_v3
```

### 18.2 복사 폴더 구조

작업폴더 아래에는 다음 구조로 복사한다.

```text
00_원본복사
 ├─ LETTER_OUT_발신공문
 ├─ LETTER_IN_수신공문
 ├─ MEETING_회의록
 ├─ PROGRESS_공정
 ├─ SAFETY_안전
 ├─ QUALITY_품질
 ├─ INSPECTION_검측
 ├─ MATERIAL_자재승인
 ├─ TECH_REVIEW_기술검토
 ├─ PAYMENT_기성
 ├─ PHOTO_사진
 ├─ DN_결함통보
 ├─ NCR_부적합
 ├─ FI_작업지시
 ├─ SCAR_안전시정
 └─ UNKNOWN_미분류
```

### 18.3 동일 파일 처리

동일 이름의 파일이 이미 있을 경우 다음 옵션을 제공한다.

| 옵션 | 설명 |
|---|---|
| 덮어쓰기 | 기존 파일을 새 파일로 교체 |
| 건너뛰기 | 기존 파일 유지 |
| 이름 변경 | 파일명 뒤에 `_copy1` 추가 |
| 해시 비교 후 처리 | 파일 내용이 동일하면 건너뛰고 다르면 이름 변경 |

### 18.4 사용자 확인 방식

동일 파일 충돌 시 다음 버튼을 제공한다.

- 이 파일만 덮어쓰기
- 이 파일만 건너뛰기
- 이 파일만 이름 변경
- 동일 작업 모두 덮어쓰기
- 동일 작업 모두 건너뛰기
- 동일 작업 모두 이름 변경
- 취소

## 19. 자동 분류 기능

### 19.1 분류 기준

자동 분류는 다음 순서로 수행한다.

1. 환경설정의 문서 유형별 폴더 키워드 확인
2. 파일명 키워드 확인
3. 파일 확장자 확인
4. 본문 텍스트 추출 후 키워드 확인
5. 샘플 문서와의 유사도 비교
6. 로컬 LLM을 통한 보조 분류
7. 확신도가 낮으면 미분류로 처리

### 19.2 분류 결과

각 파일은 다음 정보를 가진다.

```json
{
  "file_path": "D:/source/2026-07-10_발신공문.hwpx",
  "copied_path": "D:/work/2026-07/00_원본복사/LETTER_OUT_발신공문/...",
  "document_type": "LETTER_OUT",
  "confidence": 0.91,
  "matched_rules": ["filename_keyword:공문", "body_keyword:문서번호"],
  "date": "2026-07-10",
  "status": "classified"
}
```

### 19.3 낮은 신뢰도 처리

분류 신뢰도가 기준값 미만이면 다음 처리한다.

- `UNKNOWN_미분류` 폴더에 복사
- 검토 화면에서 사용자에게 분류 선택 요청
- 사용자가 수정한 분류 결과를 학습 규칙에 반영할 수 있도록 저장

5. 표 태그 위치에 대장 표 삽입
6. 사진대지 태그 위치에 이미지 삽입
7. Appendix 태그 위치에 원문 이미지 또는 캡처 삽입
8. BinData에 이미지 추가
9. manifest 및 참조 정보 갱신
10. HWPX로 재압축
11. 결과 파일 저장
12. 생성 결과 검증

- REPORT_ROUND

- DN_LIST
- NCR_LIST

- SCAR_LIST

# Part G. 비기능 요구사항

## 31. 오프라인 동작 요구사항

- 인터넷 연결 없이 실행 가능해야 한다.
- 최초 설치 후 필요한 모델과 라이브러리가 준비되어 있다면 모든 핵심 기능은 로컬에서 수행해야 한다.
- 외부 API 호출은 기본 비활성화한다.
- 외부 API 사용 옵션이 있더라도 사용 전 명시적 동의를 받아야 한다.
- 외부 API 사용 시 어떤 파일과 텍스트가 전송되는지 사전 표시해야 한다.

## 32. 보안 요구사항

- 원본 파일은 수정하지 않는다.
- 작업파일은 로컬 작업폴더에만 저장한다.
- 로그에는 개인정보와 민감정보 전문을 남기지 않는다.
- LLM 프롬프트와 응답 이력 저장 여부를 사용자가 선택할 수 있어야 한다.
- 중요 파일 삭제 시 확인 절차를 둔다.
- 외부 전송 기능은 기본 비활성화한다.

## 33. 성능 요구사항

기준 작업량:

- 문서 300개
- 이미지 500장
- 총 자료 크기 5GB

권장 성능:

- 파일 스캔: 5분 이내
- 문서 분류: 10분 이내
- 대장 추출: 15분 이내
- AI 요약: 모델 성능에 따라 5~30분
- HWPX 생성: 5분 이내

## 34. 안정성 요구사항

- 작업 중 오류가 발생해도 전체 작업이 중단되지 않아야 한다.
- 파일별 오류는 개별 기록하고 다음 파일 처리를 계속한다.
- 작업 단계별 체크포인트를 저장한다.
- 앱 종료 후 재실행 시 이전 작업을 이어서 진행할 수 있어야 한다.

## 35. 사용성 요구사항

- 신입사원도 사용할 수 있도록 단계형 화면으로 구성한다.
- 오류 메시지는 기술 용어만 표시하지 않고 원인과 조치 방법을 함께 표시한다.
- 미분류 파일과 누락 항목을 한눈에 볼 수 있어야 한다.
- 자동 생성 결과는 사용자가 수정할 수 있어야 한다.

## 40. Kiro 프로젝트 기본 원칙

Kiro로 개발할 때 다음 원칙을 적용한다.

1. 기능을 한 번에 모두 구현하지 말고 Spec 단위로 분리한다.
2. 먼저 PRD를 기반으로 상세 설계문서를 작성한다.
3. 상세 설계문서가 승인된 뒤 구현한다.
4. 구현 전 데이터 모델과 폴더 구조를 먼저 확정한다.
5. 모든 기능은 로컬 단독 실행을 기본 전제로 한다.
6. 외부 API 호출은 옵션으로만 구현하고 기본 비활성화한다.
7. 원본 파일을 수정하는 코드는 작성하지 않는다.
8. 숫자, 날짜, 문서번호는 AI 생성값을 사용하지 않는다.
9. 모든 자동 생성 결과는 사용자 검토 단계를 거친다.
10. 테스트 데이터와 샘플 템플릿을 포함한다.

11. 최종 HWPX 파일을 생성해야 한다.
12. 사용된 파일 목록과 오류 로그를 남겨야 한다.

# Part N. 개발 시 유의사항

## 51. 반드시 지켜야 할 사항

1. 원본 파일을 수정하지 않는다.
2. 모든 작업은 작업폴더 복사본 기준으로 수행한다.
3. AI가 숫자, 날짜, 문서번호, 금액을 생성하지 않도록 한다.
4. LLM 결과는 JSON으로 받고 검증 후 사용한다.
5. 사용자가 최종 생성 전 수정할 수 있도록 한다.
6. 인터넷 없는 환경을 기본 전제로 한다.
7. 로그와 오류 메시지를 충분히 남긴다.
8. HWPX 생성 실패 시 중간 데이터를 보존한다.
9. 템플릿과 코드 로직을 분리한다.
10. 설정 프로필을 내보내기/가져오기 가능하게 한다.

## 53. 최종 요약

본 애플리케이션은 CM단 월간보고서 작성 업무를 자동화하기 위한 로컬 PC 기반 도구이다.

사용자는 환경설정에서 월간보고서 양식, 취합 문서 샘플, 작업폴더, 대장 제목, 사진대지 방식, 데이터 삽입 위치, 최종 저장 규칙을 설정한다.

자동취합 단계에서는 보고년월, 취합 기준일, 원본 자료 루트경로를 입력하면 애플리케이션이 적합한 파일을 검색하고 작업폴더로 복사한 뒤, 환경설정 기준에 따라 문서를 자동 분류·추출·요약한다.

이후 사용자가 결과를 검토하고 최종 월간보고서 HWPX 파일을 생성한다.

성공적인 구현을 위해서는 LLM 성능보다 다음 요소가 더 중요하다.

- 자료 폴더 표준화
- 문서 샘플 기반 분류 기준
- HWPX 템플릿 태그 설계
- 대장 데이터 추출 정확도
- 사진대지 자동 배치
- 사용자 검토 화면
- 로컬 단독 실행 안정성

본 PRD를 기준으로 Kiro에서는 먼저 상세 설계문서를 작성하고, 기능별 Spec을 분리한 뒤, MVP부터 단계적으로 구현해야 한다.
